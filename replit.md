# Brainstorm Shell

## Overview
Brainstorm Shell is a web application built with React, TypeScript, and Vite, providing a user interface for the Brainstorm platform. It integrates Nostr login via NIP-07, offering social graph visualization, GrapeRank reputation scoring, and Web of Trust (WoT) concepts. The application aims to give users insights into their social network and influence within the Nostr ecosystem, including profile metadata fetching, social graph exploration, and NIP-85 service provider activation.

**Anonymous, search-first experience:** Logged-out visitors land on a Google-style search-first home at `/`, and can run searches and view full profile pages using the NosFabrica ("house") trust perspective with no login. Sign-in via NIP-07 stays optional and unlocks the personalized "My WoT" perspective plus account-only pages (Dashboard/Network/Settings/My Panel/Admin/Onboarding), which are hidden from anonymous visitors via a `RequireAuth` wrapper in `client/src/App.tsx` (redirect to `/`, no previews). Public routes: `/`, `/search`, `/profile/:npub`, `/faq`, `/what-is-wot`. The header on public pages shows a subtle "Sign in" button (`client/src/components/SignInButton.tsx` — NIP-07 + LoginFailureModal), and locked features (e.g. follow/mute/report on a profile) show inline sign-in prompts. Anon-viewable data calls (`getUserByPubkey/getUserOverview/getUserStats/getUserConnections`) use `optionalAuthFetch` in `client/src/services/api.ts`: it routes through `authenticatedFetch` only when a session exists (`brainstorm_session_token` OR `nostr_user`), otherwise a plain fetch with NO redirect/storage-wipe side effects — so anon 401s degrade gracefully (e.g. "Profile not found") instead of bouncing to `/`. **Backend gap:** token-less serving of house-POV profile scores (`/user/{pubkey}/overview|stats|connections`) is coordinated separately; until it lands, anonymous profile data 401s and the page shows the graceful empty state.

## System Architecture
The application uses a React 18 frontend with TypeScript, Vite, Tailwind CSS, and shadcn/ui. Routing is managed by `wouter`. An Express.js backend proxies API requests to mitigate CORS issues, especially for Nostr profile fetching and NIP-05 resolution.

**Key Architectural Decisions:**
- **UI/UX:** Features a dark-theme with glassmorphism, gradient texts, and CSS-only animations. Designs emphasize polished user experiences for login, dashboard, and other core pages, including animated graph backgrounds and custom SVG icons.
- **Nostr Integration:** Utilizes `applesauce-core` for NIP-07 login and profile handling, and `nostr-tools` for `nip19` encoding. Authentication uses a challenge-response mechanism with the Brainstorm backend via kind 22242 events.
- **Data Management:** React Query is used for data fetching, caching, and state management for user profiles, GrapeRank results, and social graph data.
- **Feature Specifications:**
    - **Authentication:** NIP-07 based challenge-response login with session tokens stored in `localStorage`. Silent re-authentication on 401 (expired token) via `silentReauth()` in `client/src/services/api.ts` — automatically re-authenticates using NIP-07 extension without user intervention on hard refresh. Falls back to login redirect if extension unavailable. All protected pages guard against error flash during auth redirects via `isAuthRedirecting()`.
    - **Dashboard:** Displays GrapeRank results, social graph counts, and an onboarding panel. Network Health pie chart includes a "Flagged" tier (red `#ef4444`) for users in the `low_and_reported_by_2_or_more_trusted_pubkeys` API field. Supported Clients carousel has 3 slides: Amethyst, Nostria, and a Developer CTA ("Get Your Client Featured") linking to `/faq?tab=developers`.
    - **Search Page:** Provides a lookup tool for `npub`, hex pubkeys, and NIP-05 handles, redirecting to the Profile page upon successful lookup. Free-text profile search uses the Brainstorm backend `GET /search/byText?text=...&onlyRanked=true` endpoint via `apiClient.searchByText()` in `client/src/services/api.ts` (replaces the legacy `brainstorm.world/api/search/profiles/meili` Meilisearch endpoint). Result mapping handles the dynamic `rank_<taPubkey>` field by picking up the first `rank_*` key on each hit (in `byTextResultToSearchResult` helper in SearchPage.tsx and `searchProfilesMeili` in `services/nostr.ts`). The new endpoint does not yet support pagination, `wot_followers`, or per-user trust perspective (POV) — the "My WoT" option in the POV dropdown is shown as disabled with a "Coming soon" label until backend support lands.
    - **Global Trust Perspective (POV):** The selected POV (NosFabrica "house" vs the logged-in user's "My WoT") is a global app-level setting persisted in `localStorage` key `brainstorm_active_pov`. Source of truth: `useActivePov()` hook in `client/src/hooks/useActivePov.ts` (with `getActivePov`/`setActivePov`/`hasStoredPov` helpers, plus same-tab `brainstorm-pov-changed` CustomEvent and cross-tab `storage` event sync). Availability of "My WoT" is gated by `useHasMywot()` in `client/src/hooks/useHasMywot.ts` (true iff `/user/self` returns a `history.ta_pubkey`). **Login default:** if the user has never picked a perspective (`!hasStoredPov()`) AND already has a calculated trust graph (`hasMywot === true`), `HeaderPovChip` auto-initializes the perspective to `"mywot"`; otherwise it falls back to `"nosfabrica"`. The desktop header no longer renders a standalone POV chip. Instead, a small colored badge is overlaid on the bottom-right of the user avatar (indigo with the NosFabrica logo when POV=nosfabrica, emerald with the user's picture/initial when POV=mywot) — see `<PovBadge user={user} />` from `client/src/components/PovBadge.tsx`. The actual switcher lives inside the avatar dropdown menu as the top section via `<PovMenuSection user={user} scope=... />`, mirroring the MobileMenu pattern so there is only ever one dropdown next to the avatar across every authenticated page (Search, Dashboard, Profile, Network, Settings, Admin, UserPanel, FAQ). The auto-default-to-mywot effect (when `!hasStoredPov && hasMywot`) lives in a single `<PovAutoDefault />` mounted once in `client/src/App.tsx` so it runs regardless of which page is active. When `!hasMywot`, the "My WoT" row in the dropdown is dimmed and shows a "Calculate yours →" deep-link to `/settings` (data-testid `link-calculate-yours`). Mobile users still see a "Trust Perspective" section at the top of the MobileMenu drawer. The Search page's previous inline POV switcher inside the search input has been removed in favor of this single global control. **Backend support gap:** today only the Search page's Meili endpoint (`apiClient.searchProfilesLegacyMeili`) actually accepts a `wotPov=house|user` parameter — `/user/{pubkey}/overview`, `/user/{pubkey}/stats`, and `/user/{pubkey}/connections` do not. So Profile/Network/Dashboard scores are always rendered from the logged-in user's personalized graph regardless of the global POV setting. The chip on those pages is passed `scope="page-not-supported"`, which surfaces an honest amber warning inside the dropdown ("Scores on this page always reflect your personalized view. House-perspective scores aren't supported by the backend here yet."). Search passes `scope="global"` because the toggle actually drives data there and defensively normalizes `mywot+!hasMywot` to `nosfabrica` before issuing the request.
    - **Profile Page (`/profile/:npub`):** Shows trust score, social reach, risk assessment, shared connections, and raw API data. Allows drill-down into expanded lists and navigation to other profiles. Risk Assessment section shows a "Flagged" indicator (red styling) when the viewed profile exists in the observer's own flagged list (`low_and_reported_by_2_or_more_trusted_pubkeys` from `/user/self`), since flagged status is relative to the observer's trust graph. **Connection pills (Option B):** Each row in the expanded Followers/Following/Mutual/Shared panels renders at most two clustered relationship pills via `deriveConnectionClusters()` in ProfilePage.tsx — a teal **owner ↔ them** pill (Mutual / Follower / Following with ↔/←/→ icons) reflecting the profile owner's relationship, and an indigo **you ↔ them** pill (Mutual with you / Follows you / You follow) reflecting the logged-in user's relationship. Replaces the previous 4-word-pill layout (Following/Mutual/Shared Follower/Shared Following) which was redundant and confusing. Mute/report pills still render as their own separate badges next to the cluster pills. The current section's own group key is still filtered out (e.g. inside the "shared_followers" panel, the "Follows you" you-cluster pill is suppressed). Test IDs: `pill-owner-<link>-<pk8>`, `pill-you-<link>-<pk8>`. Expanded panels for report/mute sections fetch metadata directly from Nostr relays on expand: kind 1984 events for reports (showing report type badge, relative timestamp, reason) and kind 10000 events for mute lists (showing mute list update timestamps). Report type badges use color-coded styles (spam=amber, impersonation=red, nudity=pink, illegal=red, profanity=orange, other=slate). **POV mitigation (frontend-only — backend doesn't accept `wotPov` on `/user/{pubkey}/overview|stats|connections`):** Search → Profile navigation appends `?pov=nosfabrica|mywot` reflecting the user's POV at click time; the Profile page reads the URL POV (preferred) or the global POV via `useActivePov()` to derive `effectivePov`, which is compared against the always-personalized `RENDERED_POV = "mywot"`. When `povMismatch === true`, a dismissable amber banner (`banner-pov-mismatch`) renders at the top of the profile header card explaining the gap and linking to a FAQ entry. The Brainstorm score badge in the header is now a **dual-meter widget (Variant B)**: when the search seed contains a NosFabrica-perspective rank (`ProfileSeed.wotRankNosfabrica`, captured from the Meili hit's `wot_rank` field), two side-by-side mini meters render — left = NosFabrica (indigo `#6366f1`, `meter-nosfabrica`), right = You (emerald `#10b981`, `meter-you` from personalized API) — with a small agreement chip below ("Both perspectives agree" / "Slight difference · N pts" / "Perspectives diverge · N pts"). When no NosFabrica seed exists (deep links without search context), it gracefully degrades to a single "You" meter (no empty cell). Tier color and confidence banner stay anchored to the personalized score — the mismatch banner carries the perspective story instead of letting colors flicker between Search and Profile.
    - **Network Page:** A CRM-style social graph explorer for managing connections by group with filtering and search. Includes a "Flagged" group (red `#ef4444`) sourced from the API field `low_and_reported_by_2_or_more_trusted_pubkeys`. Detail panel shows a "Flagged" badge when the user is in the observer's flagged list (not from per-profile API data).
    - **FAQ Page (`/faq`):** Product-specific FAQ with two-tab layout ("Using Brainstorm" for end users, "For Developers" for NIP-85 client integration). Supports `?tab=developers` query param deep-linking. Uses accordion Q&A pattern matching WhatIsWotPage style. Accessible to both authenticated and unauthenticated users. Linked from user dropdown menu (above Settings) and mobile navigation (above "What is WoT?") on all pages.
    - **What is WoT? Page:** An educational resource with interactive trust scenarios.
    - **NIP-85 Service Provider Activation/Deactivation:** A guided process for activating Brainstorm as a NIP-85 service provider (Kind 10040 event). Users can also deactivate from the Settings page, which publishes a replacement Kind 10040 event with empty tags to relays, clearing the provider declaration. Deactivation clears `brainstorm_nip85_activated` from localStorage and reloads the page. Functions: `signNip85()` and `signNip85Deactivation()` in `client/src/services/nostr.ts`.
    - **Settings Page:** Manages NIP-85 status (activate/deactivate/republish), GrapeRank calculation triggers, and trust perspective settings. NIP-85 and GrapeRank buttons are disabled when `hasNoFollowing` is true (user has no follows). Trust Perspective presets (Relax/Default/Strict) are functional — each sets a verified threshold (0.00/0.02/0.15) stored in `localStorage` key `brainstorm_trust_preset`, which controls "verified" vs "unverified" display across Dashboard, Network, and Profile pages. Utility module: `client/src/services/trustThreshold.ts`. Includes a "Contact & Support" section at the bottom with two cards: "List Your Client" (developer outreach for NIP-85 client listings) and "Get in Touch" (general support), both linking to `support@nosfabrica.com`.
    - **Admin Dashboard (`/admin`):** NosFabrica Macro Admin Dashboard accessible only to whitelisted pubkeys (configured in `client/src/config/adminAccess.ts`). Features: KPI cards strip (Total Pubkeys, Followers, Following, Queue Depth, Reports Filed), tab-based navigation (Overview, Activity, Users, Assistants, System Health), sortable/searchable/paginated user directory from graph data, GrapeRank pipeline status, relay and API health monitoring, infrastructure overview, and activity log. The "Assistants" tab surfaces Brainstorm Assistant publish metrics (total assistants, total publishes, last 24h/7d, last publish time) and a per-user publish history table backed by `/admin/assistants/stats`, `/admin/assistants`, and `/admin/assistants/:pubkey/history`. Unsupported admin API endpoints show "Not Connected" status badges. Admin link appears in profile dropdown menus (amber-styled) and MobileMenu only for whitelisted pubkeys. Non-admin users are redirected to `/dashboard`. Expanding any user row in the Users tab renders a **Nostr Health** card (`client/src/components/admin/NostrHealthCard.tsx`) under the Identity strip with two live-from-relays panels: Trust Provider List (kind 10040) with per-tag checks (event present, `30382:rank`/`30382:followers` tags, inner pubkey matches assigned TA, relay hint matches `VITE_NIP85_RELAY_URL`) using existential ("any matching tag wins") semantics matching `isUsingBrainstorm`, and Profile Metadata (kind 0) with avatar/name/nip05/about plus a Brainstorm Assistant detection badge (green when the `website` field references a known Brainstorm host) and an "Assigned Brainstorm Assistant" sub-section that fetches the assistant's own kind 0 (resolved from admin-assigned `taPubkey`, falling back to `30382:rank`/`30382:followers` inner pubkey from kind 10040; "Self-assigned" badge when the assistant pubkey equals the user). Both panels include collapsible raw-JSON viewers with copy buttons. When a user has multiple `30382:rank` or `30382:followers` tags published in their kind 10040, an "All published tags" disclosure surfaces every tag with per-tag pubkey/relay pass-fail badges (winner = green "Active", non-matching = amber "Stale", non-matching but harmless = neutral "Duplicate"); the overall status pill bumps to amber "Issues found" when stale duplicates exist so admins can detect leftover/old tags from prior activations. Backed by `checkNip85Health()` in `client/src/services/nostr.ts` (`Nip85TagDetail` type, `rankTags`/`followersTags` arrays).
    - **My Panel Page (`/panel`):** Web of Trust management center for logged-in users. Features: Account Overview header strip (pubkey, NIP-85 status, network size, last calc timestamp), Brainstorm Assistant Profile publishing (kind 0 event with website field — calls backend endpoint with graceful "Coming soon" fallback if 404), Invite & Add Users section (shareable invite link + add-by-npub lookup), Network Score Monitor (sortable/searchable table of followed accounts with trust scores, tier badges, and trend indicators comparing to locally cached previous scores), Trust Attestation History (chronological log of TA publish events with timestamps, event kind, relays, success/failure status), and DCoSL Lists scaffold (Coming Soon placeholder). API method: `apiClient.publishBrainstormAssistantProfile()` in `client/src/services/api.ts`.
    - **Dashboard Empty State:** When users have no follows (`hasNoFollowing`), the dashboard hides the Network Health section and recalculation panel, showing only the welcome header with "Set up your trust network" subtitle and the "No follows yet" banner. Trust Signals badge shows step-specific status during calculations: "Calculating…" → "Publishing…" → "Complete".
- **Performance Optimizations:** Includes 60-second timeouts for server proxy routes, fetches the logged-in user's profile metadata (kind 0) immediately at login (fire-and-forget in `completeLogin` in `client/src/services/nostr.ts`) so the avatar/name appear sooner instead of waiting for the dashboard to mount — `setCurrentUser` dispatches the `brainstorm-user-changed` event on profile-field changes (not just pubkey changes) so the header/menu re-render reactively, and the dashboard's `profileQuery` stays gated on missing name/picture so it skips the duplicate fetch once login populates them. Uses branded loading animations. Network page search uses on-demand debounced profile fetching (500ms delay, batches of 50, max 500 uncached profiles) instead of eager bulk loading — keeps initial page load fast while enabling name-based search across the full group.

## External Dependencies
- **Nostr Protocol:** Interacts with various Nostr relays (e.g., damus, nostr.band, nos.lol) for metadata fetching and event publishing.
- **Brainstorm Backend API:** Switchable between Staging (`brainstormserver-staging.nosfabrica.com`) and Production (`brainstormserver.nosfabrica.com`) via admin environment selector. Selection persisted in `localStorage` key `brainstorm_api_env`. Dynamic URL resolution in `client/src/services/api.ts` via `getApiEnvironment()`/`setApiEnvironment()`/`getApiBaseUrl()`. Admin dashboard shows environment badge and requires confirmation dialog before switching (with extra warning for Production).
- **Nostr HTTP APIs:** `api.nostr.band` and `purplepag.es` serve as HTTP fallback for profile fetching.
- **NIP-07 Browser Extension:** Required for Nostr key management and event signing.

## Shared Components
- **MobileMenu (`client/src/components/MobileMenu.tsx`):** Unified mobile navigation drawer used by all authenticated pages (Dashboard, Settings, Network, Search, Profile, FAQ, Admin, My Panel). Organized into grouped sections: Navigation (Dashboard/Search/Network/My Panel), Help & Info (FAQ/What is WoT?), Account (Settings + Admin for whitelisted pubkeys). Navigation and Account sections are auth-gated (hidden when user is null). Network button disabled when `calcDone` is false. Admin nav only visible when `isAdmin` prop is true. User avatar + Sign Out in footer.
